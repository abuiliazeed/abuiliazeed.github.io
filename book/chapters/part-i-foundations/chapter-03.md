# Chapter 3: The Empty Repository — Starting from Zero

> *"The first commit to an empty repository landed in late August 2025. There was no pre-existing human-written code to anchor the system. From the beginning, the repository was shaped by the agent."*
> — OpenAI Harness Engineering Blog Post, February 2026¹

---

There's a moment in every software project that's simultaneously terrifying and exhilarating: the empty repository. No code. No structure. No history. Just a blank canvas and infinite possibility. Most engineers have felt it — that brief paralysis before the first commit, where every decision feels consequential because there's nothing to constrain your options.

Now imagine handing that blank canvas to an AI agent.

The OpenAI team did exactly this. Their first commit — repository structure, CI configuration, formatting rules, package manager setup, and application framework — was generated by Codex CLI using GPT-5, guided by a small set of existing templates. Even the initial AGENTS.md was written by the agent itself. The repository was shaped, from its very first moment, by the agent's understanding of what a well-structured codebase looks like.

This wasn't an accident. It was a deliberate choice, and understanding *why* they made it — and what advantages it gave them — is essential for anyone building with agents.

## Why Start from Nothing?

Starting from an empty repository is the purest expression of agent-first development. When you begin with nothing, you're forced to build everything the agent needs from the ground up — the structure, the constraints, the documentation, the tooling. There's no legacy code to work around, no inconsistencies inherited from years of human decisions, no "we've always done it this way" baggage.

This is a feature, not a bug.

Here's why: existing codebases are designed for humans. Their structure, naming conventions, documentation patterns, and architectural decisions were made with human cognition in mind — not agent cognition. When you drop an agent into an existing codebase, it's like asking someone to navigate a city that was built for a different species. The signs are in the wrong language, the roads don't go where you expect, and the landmarks don't make sense.

When you start from nothing, you can build the city for its inhabitants. The agent becomes the primary "reader" of the codebase, and you design everything — the directory structure, the documentation, the naming conventions, the architectural patterns — for agent legibility first and human legibility second.

The OpenAI team was explicit about this priority:

> *"Because the repository is entirely agent-generated, it's optimized first for Codex's legibility. In the same way teams aim to improve navigability of their code for new engineering hires, our human engineers' goal was making it possible for an agent to reason about the full business domain directly from the repository itself."*

Starting from zero also eliminates the hardest problem in brownfield agent development: *context contamination*. In an existing codebase, the agent sees years of accumulated decisions — some good, some bad, some incomprehensible. It learns from what it sees, including the bad patterns. An agent working in a messy codebase will produce messy code because that's what "normal" looks like in that codebase. It doesn't know that the 500-line utility function is an anti-pattern — it thinks that's just how functions are written here.

A greenfield codebase has no bad patterns to learn from because it has no patterns at all. Every pattern is intentionally established by you, through the harness, before the agent writes its first line of production code.

## The First Steps (Architecture → Skeleton → Infrastructure → Thin Slice → Iterate)

The path from an empty repository to a working product follows a specific sequence. This isn't arbitrary — it's the order that maximizes the agent's ability to build on each successive layer.

### Step 1: Architecture

Before any code is written, you need to define the architecture. Not in the abstract — in concrete, mechanizable terms. What are the layers? What are the dependency directions? What are the naming conventions? What are the boundaries between modules?

The OpenAI team defined a rigid layer architecture early:

```
Types → Config → Data (Repo) → Services → Runtime → UI
```

Each business domain follows this layer structure, and dependencies can only flow "forward" through the layers. Cross-cutting concerns (auth, connectors, telemetry, feature flags) enter through a single explicit interface called Providers. Anything else is disallowed.

This architecture was defined before the first line of application code. Not because they had perfect foresight, but because the architecture *is* the harness. Without it, the agent has no boundaries to work within, and every subsequent line of code is an architectural gamble.

Your architecture document should answer these questions:

- **What are the major modules/domains?** (e.g., User Management, Payment Processing, Notification Service)
- **What are the layers within each module?** (e.g., Types, Config, Data, Services, Runtime, UI)
- **What are the dependency direction rules?** (e.g., "Types can depend on nothing. Services can depend on Types and Data. UI can depend on everything.")
- **What are the cross-cutting concerns?** (e.g., Auth, Logging, Feature Flags)
- **How do cross-cutting concerns interact with modules?** (e.g., "Through a Provider interface")
- **What technology choices have been made?** (e.g., "TypeScript for all code, Zod for schema validation, React for UI")

Don't aim for perfection. Aim for clarity. The architecture will evolve, but the initial version needs to be clear enough that you can encode it in linters and structural tests.

### Step 2: Skeleton

Once the architecture is defined, generate the repository skeleton. This includes:

- **Directory structure** matching the architecture
- **Package manager configuration** (e.g., `package.json`, `pyproject.toml`)
- **Build system configuration** (e.g., `tsconfig.json`, webpack config)
- **CI pipeline configuration** (e.g., GitHub Actions workflow)
- **Formatting rules** (e.g., `.prettierrc`, `.eslintrc.js`)
- **Initial AGENTS.md** with basic instructions and pointers to architecture docs

The OpenAI team generated this skeleton using Codex CLI guided by templates. The important thing is that the skeleton establishes the *shape* of the codebase before any application logic exists. When the agent starts writing application code, it's filling in a well-defined structure, not inventing one.

Here's what a minimal skeleton might look like:

```
my-project/
├── AGENTS.md                    # Agent instructions (table of contents)
├── ARCHITECTURE.md              # Architecture definition
├── package.json                 # Dependencies and scripts
├── tsconfig.json                # TypeScript configuration
├── .eslintrc.js                 # Lint rules (including custom)
├── .prettierrc                  # Formatting rules
├── .github/
│   └── workflows/
│       └── ci.yml               # CI pipeline
├── src/
│   ├── types/                   # Type definitions
│   ├── config/                  # Configuration
│   ├── data/                    # Data access layer
│   ├── services/                # Business logic
│   ├── runtime/                 # Runtime / server
│   └── ui/                      # User interface
├── tests/
│   ├── structural/              # Architectural invariant tests
│   └── unit/                    # Unit tests
└── docs/
    ├── design-docs/
    ├── product-specs/
    └── exec-plans/
```

Notice that the skeleton includes both code structure (`src/`) and harness structure (`docs/`, `tests/structural/`, custom lint rules). The harness exists before the application does.

### Step 3: Infrastructure

With the skeleton in place, build the infrastructure layer. This is the tooling and automation that makes the harness operational:

- **Custom linters** that enforce the architectural rules
- **Structural tests** that verify architectural invariants
- **CI pipeline** that runs linters and tests on every PR
- **Local development environment** that agents can boot and interact with
- **Observability stack** (even if minimal at this stage)

The OpenAI team's custom linters were themselves generated by Codex. This is a beautiful self-reinforcing loop: the agent builds the tools that constrain its own behavior. It's like asking a contractor to build the scaffolding before they build the building — the scaffolding ensures the building goes up straight, and the contractor is the most qualified person to design scaffolding they'll actually use.

Here's a simple example of a custom lint rule that enforces dependency direction:

```javascript
// lint-rules/dependency-direction.js
// Enforces that 'types' layer cannot import from 'services' layer

module.exports = {
  meta: {
    type: 'problem',
    docs: {
      description: 'Enforce dependency direction rules',
      category: 'Architecture',
    },
    messages: {
      invalidDependency:
        'Layer "{{fromLayer}}" cannot import from "{{toLayer}}". ' +
        'Dependencies must flow forward: Types → Config → Data → Services → Runtime → UI. ' +
        'Move shared types to the Types layer or restructure the dependency.',
    },
  },
  create(context) {
    const layerOrder = ['types', 'config', 'data', 'services', 'runtime', 'ui'];
    const filePath = context.getFilename();

    // Determine which layer this file belongs to
    const currentLayer = layerOrder.find(layer =>
      filePath.includes(`/src/${layer}/`)
    );

    if (!currentLayer) return {};

    return {
      ImportDeclaration(node) {
        const importPath = node.source.value;
        const importLayer = layerOrder.find(layer =>
          importPath.includes(`/src/${layer}/`)
        );

        if (!importLayer) return;

        const currentIndex = layerOrder.indexOf(currentLayer);
        const importIndex = layerOrder.indexOf(importLayer);

        if (importIndex > currentIndex) {
          context.report({
            node,
            messageId: 'invalidDependency',
            data: {
              fromLayer: currentLayer,
              toLayer: importLayer,
            },
          });
        }
      },
    };
  },
};
```

Here's the same dependency-direction rule implemented as a Python AST-based linter:

```python
# lint_rules/dependency_direction.py
"""Enforces that 'types' layer cannot import from 'services' layer."""
import ast
from pathlib import Path

LAYER_ORDER = ['types', 'config', 'data', 'services', 'runtime', 'ui']

def check_dependency_direction(filepath: str) -> list[str]:
    """Check a Python file for invalid layer dependencies."""
    violations = []
    path = Path(filepath)
    current_layer = next(
        (l for l in LAYER_ORDER if f'/src/{l}/' in str(path)), None
    )
    if not current_layer:
        return violations

    tree = ast.parse(path.read_text())
    for node in ast.walk(tree):
        if isinstance(node, (ast.Import, ast.ImportFrom)):
            module = (
                node.module or ''
                if isinstance(node, ast.ImportFrom)
                else node.names[0].name
            )
            import_layer = next(
                (l for l in LAYER_ORDER if f'/src/{l}/' in module), None
            )
            if import_layer and LAYER_ORDER.index(import_layer) > LAYER_ORDER.index(current_layer):
                violations.append(
                    f"Layer '{current_layer}' cannot import from "
                    f"'{import_layer}'. Dependencies must flow forward: "
                    f"Types → Config → Data → Services → Runtime → UI. "
                    f"Move shared types to the Types layer or restructure the dependency."
                )
    return violations
```

Notice the error message. It doesn't just say "invalid dependency." It explains the rule (dependencies flow forward through layers), identifies the specific violation, and suggests a remediation path. This is an *agent-readable error message* — designed to help the agent fix the problem autonomously.

This kind of linter is the infrastructure that the OpenAI team built before the first feature. The agents wrote the linters (yes, the agents built their own guardrails), and then those linters governed all subsequent agent work. It's a self-reinforcing system: the better the infrastructure, the better the agent output; the better the agent output, the more reliable the infrastructure.

### Step 4: Thin Slice

With the skeleton and infrastructure in place, it's time to build the first thin slice of functionality. This is a minimal end-to-end feature that exercises every layer of the architecture — from types through UI — but does so in the simplest possible way.

Why a thin slice? Because it validates the architecture. If the thin slice is awkward to implement within the architectural constraints, the architecture needs adjusting. Better to discover this early, when you have ten files, than later, when you have ten thousand.

The thin slice also validates the *harness*. Can the agent work within the constraints? Do the linters catch the right things? Is the documentation sufficient? Does the CI pipeline provide useful feedback? The thin slice is a test of the entire system — not just the application, but the development environment.

Choose a thin slice that:

- Crosses all architectural layers
- Involves real data flow (not just "hello world")
- Requires meaningful testing
- Exposes the agent to every tool in the harness

For a user management system, a good thin slice might be: "A user can register with an email and password, and receives a welcome email." This involves types (user schema), data (persistence), services (registration logic, email sending), runtime (API endpoint), and UI (registration form). It's simple but complete.

### Step 5: Iterate

With the thin slice validated, you iterate. Each iteration adds functionality, and each addition exercises the harness. The pattern is:

1. **Describe the task** (what to build)
2. **Let the agent implement it** (within the harness constraints)
3. **Observe the result** (does it work? is it correct? does it follow the architecture?)
4. **Adjust the harness** (add constraints, improve context, enhance verification)
5. **Repeat**

The OpenAI team described this as working "depth-first": *"breaking down larger goals into smaller building blocks (design, code, review, test, etc.), prompting the agent to construct those blocks, and using them to unlock more complex tasks."*

This iterative pattern is crucial. The harness is not static — it grows with each iteration. Every agent mistake that makes it through the current constraints is a signal to extend the harness. Over time, the harness becomes more comprehensive, the agent becomes more autonomous, and the human's role shifts from guiding implementation to setting direction and verifying outcomes.

Let me illustrate this with a concrete example from the OpenAI experience. In the early weeks, the team noticed that agents were using `console.log` for debugging instead of the structured logging system they'd designed. Each time, they'd manually fix it in the PR. Then they had the insight: instead of fixing it each time, write a linter that catches it.

The linter took an hour to write (the agent wrote it). It has saved hundreds of hours since. Every future PR that tries to use `console.log` is caught automatically, and the error message tells the agent exactly what to use instead. One observation → one linter → permanent fix.

This is the "observe → name → document → mechanize → verify" cycle in action. And it's why agent-first teams get more productive over time: every mistake makes the harness stronger, and a stronger harness means fewer mistakes next time. The compound effect is extraordinary.

### The Progressive Autonomy Curve

As you iterate, something important happens: the agent's effective autonomy increases. Not because the model got smarter (though it might have), but because the harness got better. More constraints mean fewer wrong paths. Better context means better decisions. Stronger verification means faster self-correction.

The OpenAI team documented this progression explicitly. Their repository recently crossed a meaningful threshold where Codex can end-to-end drive a new feature. Given a single prompt, the agent can now:

1. Validate the current state of the codebase
2. Reproduce a reported bug
3. Record a video demonstrating the failure
4. Implement a fix
5. Validate the fix by driving the application
6. Record a second video demonstrating the resolution
7. Open a pull request
8. Respond to agent and human feedback
9. Detect and remediate build failures
10. Escalate to a human only when judgment is required
11. Merge the change

This progression — from needing human guidance for every step to being able to execute an entire feature lifecycle autonomously — is the result of sustained harness investment, not model improvement. The model is the same. The harness is what changed.

This is the "trust gradient" that every agent-first team climbs. You start with high human involvement (specify every step, review every line) and gradually decrease involvement as the harness proves reliable. The gradient is not linear — it accelerates as the harness improves, because each improvement enables more autonomy, which generates more data for further improvement.

The Affirm team is earlier in this curve. Their workflow still has explicit human checkpoints at every major decision point. But their trajectory is the same: as their harness matures (better CI, better documentation, better verification), they'll be able to remove checkpoints and increase autonomy.

Importantly, this autonomy increase is not a binary switch — it's a spectrum. Different tasks have different autonomy levels. A well-understood bug fix in a mature module might run at near-full autonomy. A new feature in an unfamiliar area might require significant human guidance. The harness should support this variance — not force every task to the same autonomy level.

## The Advantage of Greenfield Agent Development

Let me be explicit about why starting greenfield is advantageous, because it's not just about avoiding legacy code.

**Advantage 1: Clean pattern establishment.** In a greenfield project, every pattern is established intentionally. There are no "accidental patterns" for the agent to learn from. When the first service is written with proper dependency injection, structured logging, and comprehensive tests, the agent sees this as the *canonical pattern* and replicates it. The first examples set the tone for everything that follows.

**Advantage 2: Harness-first architecture.** You can design the architecture around the harness, rather than retrofitting a harness onto an existing architecture. The OpenAI team's layer architecture was designed specifically to be enforceable by linters — something that's much harder to do with an existing codebase that may have evolved organically.

**Advantage 3: Documentation from day one.** In a greenfield project, documentation is written before (or alongside) the code. There's no "we'll document it later" phase because the documentation *is* the context that the agent needs to write the code. This creates a virtuous cycle: good documentation produces good code, which produces more context for better documentation.

**Advantage 4: No coordination debt.** Existing codebases accumulate what I call "coordination debt" — the implicit knowledge that exists in people's heads about how the system works. "Oh, you can't change that endpoint because the mobile team depends on its current behavior." "Don't touch the payment module — it has special error handling that's not documented anywhere." This invisible knowledge is inaccessible to agents. A greenfield project has no coordination debt because there's nothing to be coordinated yet.

**Advantage 5: Progressive complexity.** You can add complexity gradually, and the harness grows with it. Instead of confronting the agent with a massive existing codebase all at once, you build the codebase incrementally. Each addition is small enough that the agent can understand it in context, and the harness captures the rationale for each architectural decision as it's made.

**Advantage 6: The "agent-wrote-it-so-the-agent-can-read-it" principle.** When code is generated by an agent, it tends to be more uniform and predictable than human-written code. Different humans have different styles, preferences, and levels of experience. Different agent sessions with the same model, same constraints, and same context produce much more consistent output. This consistency makes the codebase more legible to future agent sessions, creating a positive feedback loop.

### Self-Review and Agent-to-Agent Review

The OpenAI team developed a powerful pattern for agent quality assurance: the agent reviews its own changes, requests additional agent reviews, and iterates in a loop until all reviewers are satisfied. This self-reinforcing quality loop — which we'll explore in detail as the **Ralph Wiggum Loop** in Chapter 14 — is what enables agent review to scale in ways human review cannot.

The key insight is that you can have multiple agent reviewers examining a PR simultaneously, each bringing a different perspective (security, performance, architecture, style), with the review completing in minutes rather than days. Over time, the OpenAI team pushed almost all review effort toward agent-to-agent review. Humans may still review PRs, but they aren't required to — the harness provides sufficient quality assurance for most changes.

> **When NOT to Use Agents**
>
> Not everything needs a harness. Agent-first development is powerful, but it's not universally applicable. Understanding when *not* to use agents is as important as knowing when to use them.
>
> **Prototyping and throwaway experiments.** If you're exploring an idea that might be discarded tomorrow, the investment in AGENTS.md, linters, and CI gates is wasted. A quick prototype in a REPL or a single-file script doesn't need a harness. Use agents freely for these, but don't build infrastructure around throwaway code.
>
> **Codebases with a lifespan under one month.** If the project is a temporary bridge, a one-time migration script, or a proof-of-concept that won't ship to production, the harness overhead exceeds the benefit. The harness pays for itself over time — it needs time to pay off.
>
> **Domains with extreme regulatory requirements.** Medical device firmware, aviation control systems, and safety-critical embedded software often require documented human decision-making at every step. The regulatory frameworks governing these domains (DO-178C, IEC 62304, ISO 26262) were not designed with AI-generated code in mind. Until regulators catch up, the compliance overhead of using agents in these domains may outweigh the productivity gains. This will change — but it hasn't changed yet.
>
> **Tasks requiring deep creative intuition.** Novel algorithm design, fundamental UX innovation, and architectural breakthroughs still benefit from human creativity in ways that structured agent workflows don't capture. Agents excel at implementing known patterns; they struggle with inventing new ones. When the task is "design an algorithm nobody has designed before," the human is still the right tool for the job.
>
> The honest assessment: agent-first development is a spectrum, not a binary. Most projects benefit from *some* agent involvement. But the depth of the harness investment should match the project's expected lifespan, criticality, and creative demands. Over-investing in a harness for a throwaway project is as wasteful as under-investing in one for a production system.

## What If You Can't Start from Zero? (Brownfield Migration)

Most of you reading this are not starting from an empty repository. You have an existing codebase — maybe years old, maybe hundreds of thousands of lines, maybe worked on by dozens of engineers over time. You can't just throw it away and start over.

That's fine. The principles of harness engineering still apply. But the approach is different, and there are additional challenges.

### The Brownfield Challenge

The fundamental challenge of brownfield agent development is that existing codebases are *illegible* to agents in ways they're not illegible to humans. A human engineer can look at a 500-line function and think "this is a mess, but I can see what it does." An agent looks at the same function and sees... a 500-line function. It doesn't have the human's ability to separate signal from noise, to identify the important parts and ignore the rest, or to draw on years of domain knowledge that's not captured anywhere in the code.

This illegibility manifests in several specific ways:

**Inconsistent patterns.** Different parts of the codebase were written by different people at different times with different standards. The agent sees inconsistency and often produces inconsistent output — matching the style of whatever code it's currently working near. One module might use callbacks, another async/await, and a third might use promises with .then() chains. The agent doesn't know which is "right" — it just knows that all three are present.

**Missing documentation.** Most existing codebases have documentation that ranges from incomplete to nonexistent. Critical architectural decisions are captured in forgotten Slack threads, Jira tickets, or the memories of senior engineers. The agent can't access any of this. When the OpenAI team says "knowledge that lives in Google Docs, chat threads, or people's heads is not accessible to the system," they're describing a greenfield advantage that brownfield projects must actively overcome.

**Implicit conventions.** Every codebase has unwritten rules: "We always handle null checks this way." "Error responses use this format." "The payment module has special logging." These conventions are invisible to the agent until they're documented or encoded in tooling.

**Architectural drift.** Over time, codebases drift from their intended architecture. The "service layer" might have some files that directly access the database. The "API layer" might have business logic mixed in. The agent sees the drift and treats it as normal, perpetuating and often amplifying it. If 30% of "service" files directly query the database, the agent learns this is acceptable and does it more.

This last point is particularly insidious. In a brownfield codebase, the agent doesn't just inherit existing bad patterns — it amplifies them. Each agent session that sees and replicates a bad pattern makes that pattern slightly more prevalent, which makes the next agent session slightly more likely to replicate it. It's a positive feedback loop in the wrong direction. This is why the "correct" pillar of harness engineering — particularly automated garbage collection — is so important in brownfield projects.

### The Brownfield Migration Strategy

If you can't start from zero, here's a practical approach to migrating an existing codebase to agent-first development:

**Phase 1: Document the current state.** Before you can harness an agent, you need to understand what you have. Write an AGENTS.md that describes the codebase as it actually is — not as you wish it were. Document the directory structure, the major modules, the key patterns, and the known inconsistencies. Be honest. The agent will discover the truth anyway.

**Phase 2: Establish constraints for new code.** You can't retroactively apply constraints to existing code — that's a massive refactoring project. But you *can* apply constraints to new code. Set up linters, CI gates, and structural tests that apply to new files and new changes. Existing violations can be tracked as technical debt.

**Phase 3: Create safe zones.** Identify modules or subsystems that are relatively isolated and can be refactored independently. Make these your "agent-first zones" — areas where the agent can work with full harness support. As the agent proves itself in these zones, expand them.

**Phase 4: Incremental documentation.** As the agent works on different parts of the codebase, it will encounter gaps in documentation. Each gap is an opportunity to document something new. Over time, the documentation coverage increases, and the agent becomes more effective across more of the codebase.

**Phase 5: Progressive constraint tightening.** Start with permissive constraints and tighten them over time. Initially, linters might warn but not block. As the team adjusts, switch warnings to errors. As errors become rare, add new constraints. This progressive approach avoids the whiplash of applying strict constraints to a codebase that was built without them.

The Affirm team followed a version of this pattern. Their codebase is a twelve-year-old monorepo with structural bottlenecks: bloated test suites, manual code review, unstable CI, and deploy infrastructure not designed for agent throughput. They couldn't start from zero. Instead, they focused on building the *context files* — conventions, domain knowledge, and team decisions — that agents could find and use. They distributed tooling through an internal plugin with a central marketplace where teams could build and share their own skills. And they made explicit the workflow: Plan, Review, Execute, Verify, Review, Deliver.

### The Affirm Brownfield Experience

The Affirm case study is worth examining in detail because it's the largest documented brownfield migration to agent-first development. Their codebase is a twelve-year-old monorepo — the kind of system that makes architects wake up in a cold sweat. Their challenges were specific and instructive:

**Change review was the #1 bottleneck.** Roughly 40% of engineers in their survey raised code review as the primary friction point unprompted. PRs would sit in review for days. When you're generating agent-assisted PRs at volume, a review process designed for human-paced development becomes a chokepoint.

**CI was not built for agent throughput.** Their unit test suite at p75 ran in ~8 minutes, but full end-to-end regression suites on ephemeral test environments took 100+ minutes. The change-validate-fix-revalidate loops that agent development demands were excruciatingly slow. In one case weeks after retooling, the load from concurrent agent-submitted PRs brought down their internal test quarantine service and CI pipeline.

**Documentation was fragmented.** Over a decade of technical and product specs spread across multiple platforms. That fragmentation is "somewhat navigable for humans," as Affirm's Daniel Martin put it, but agents need clear, consolidated context. Some platforms had MCP (Model Context Protocol) integrations, others didn't, and the experience was inconsistent enough that engineers frequently had to bridge the gaps manually.

**Tool integrations created unexpected friction.** Engineers wanted to connect agents to internal systems through dozens of MCPs. But each new integration expanded the security surface in ways that required careful evaluation. The volume of requests overwhelmed the review process. CLIs — where available — proved more reliable than MCPs for many of the same tasks. But without standardized configuration, ownership, and service level agreements, the integrations that were supposed to make agents more capable became a drag.

Affirm's response to these challenges reveals the brownfield playbook:³

1. **Centralize context.** Their Architecture group is leading the effort to ensure architectural decisions, domain context, and best practices are stored where agents can find them *before* implementation starts.
2. **Permanent enablement team.** They turned the sprint team that ran retooling week into a permanent team, managing integrations, improving tool discoverability, building a support model, and tracking spend.
3. **Agent-friendly CI.** The largest investment. Automating failure classification (was it the code, a flaky test, or infrastructure?), maturing local testing to give signal before hitting CI, and right-sizing test suites.
4. **Independent verification.** Piloting a system that cross-references PR diffs against acceptance criteria using multiple models — because when an agent generates both implementation and tests, a misunderstanding can produce code and tests that confirm each other's errors.

The Affirm experience proves that brownfield migration works. But it also shows that the investment is substantial. Their "three areas of focus" — context, enablement, and validation — are ongoing programs, not one-time projects. The brownfield migration is a journey measured in months and years, not days.

### Brownfield On-Ramp: The Strangler Fig Pattern

For teams with existing codebases, a gradual migration strategy inspired by the "strangler fig" pattern from evolutionary architecture offers a pragmatic path. Rather than trying to introduce a full harness overnight, you grow it incrementally from the edges inward.

**Step 1: Introduce AGENTS.md for one service.** Pick a single, relatively isolated service or module — not the most critical one, but one that's actively developed. Write an AGENTS.md that documents its conventions, build commands, and testing requirements. Let the agent work within this small, well-defined boundary.

**Step 2: Add linter rules in "warn" mode.** Start with three to five rules targeting the most common failure patterns you observe. Don't block CI yet — just collect data on how many violations occur and where. This gives you a baseline for measuring improvement.

**Step 3: Introduce structural tests for new code only.** Write architectural invariant tests that apply to new files and modifications, not the existing codebase. This lets you enforce standards going forward without triggering hundreds of failures on legacy code.

**Step 4: Gradually increase enforcement scope.** After two to four weeks of warning-mode linters, promote the rules to errors for the services that have been migrated. Expand the enforcement perimeter one module at a time. Each module that's brought into the harness becomes an "agent-first zone" where the agent can work with full support.

Shopify followed a version of this approach with their AI Toolkit.² Rather than requiring teams to restructure their entire workflow, they built plugins that integrate into existing developer workflows — editor extensions, CI integrations, and configuration templates that work within the tools teams already use. The key insight: meet teams where they are, then progressively raise the bar as the harness proves its value.

> **⚠️ The Brownfield Trap**
> The most common failure mode in brownfield migration is trying to do too much too fast. If you apply strict linter rules to a legacy codebase all at once, you'll get hundreds of failures, the team will lose confidence, and the initiative will stall. Start narrow, measure, then expand.

## The Metaphor: Building a Factory

I want to return to the factory metaphor from the beginning of Chapter 2, because it captures the spirit of greenfield agent development perfectly.

When you build a factory, you don't start by producing widgets. You start by designing the production line: the conveyor systems, the quality checkpoints, the safety guards, the maintenance schedules, the training protocols. Only after the factory is built do you start producing widgets. And the quality of the widgets depends far more on the quality of the factory than on the skill of any individual machine.

The OpenAI team understood this intuitively. Their first weeks weren't spent building features — they were spent building the factory. The linters, the structural tests, the documentation structure, the CI pipeline, the worktree isolation, the observability stack — all of this was factory-building. And the "machines" (Codex agents) couldn't produce good widgets until the factory was properly set up.

This is why early progress was slower than expected:

> *"Early progress was slower than we expected, not because Codex was incapable, but because the environment was underspecified. The agent lacked the tools, abstractions, and internal structure required to make progress toward high-level goals."*

The environment was underspecified because the factory wasn't built yet. Once the factory was in place — once the harness had the right constraints, the right context, the right verification mechanisms — the machines ran at full speed.

Here's the critical lesson: **invest in the factory before you invest in production.** The temptation is to jump straight to features — to start the agent working on application code immediately. Resist this temptation. Spend the first week or two building the harness. The linters, the docs, the CI pipeline, the structural tests. It feels slow. It feels like you're not making progress on the product. But you're making progress on something more important: the system that will produce the product.

In a traditional development workflow, this kind of upfront investment often doesn't pay off — the project might be cancelled, the requirements might change, or the architecture might need to evolve. But in an agent-first workflow, the factory pays for itself quickly because the machines run so much faster than human developers. A week of factory-building enables months of high-throughput production.

The OpenAI team's numbers tell the story: roughly five months from empty repository to a million-line product. The first month was mostly factory-building. The last four months were high-throughput production. The ROI on that initial factory investment was extraordinary.

### When the Factory Needs Renovation

Factories need maintenance and renovation. So do harnesses. The OpenAI team's harness evolved significantly over five months:

- **Month 1:** Basic scaffolding, initial linters, architecture docs
- **Month 2:** Execution plans, quality scoring, initial garbage collection
- **Month 3:** Application legibility, UI testing via Chrome DevTools
- **Month 4:** Full observability stack (logs, metrics, traces per worktree)
- **Month 5:** Progressive autonomy, end-to-end feature driving, video recording

Each evolution was driven by a specific bottleneck. When human QA became the limiting factor, they made the application legible. When codebase entropy started accumulating, they built garbage collection agents. When agents couldn't verify UI changes, they wired in Chrome DevTools.

The factory metaphor also explains the garbage collection pattern (covered in depth in Chapter 19). In a real factory, there's waste — scrap material, misaligned components, machines that drift out of calibration. You can't prevent all waste, but you can clean it up regularly before it accumulates. The OpenAI team's garbage collection agents are the software equivalent: automated cleanup on a regular cadence, keeping the codebase clean without human intervention.

This is the factory principle in action: automate the maintenance. The factory should maintain itself as much as possible, because the humans are occupied with higher-level tasks.

## Chapter Summary

- Starting from an empty repository is the purest expression of agent-first development — you build the codebase from day one with agent legibility as the primary design constraint.
- The sequence is: Architecture → Skeleton → Infrastructure → Thin Slice → Iterate. Each step builds on the last, and the harness exists before the application does.
- Greenfield development has six key advantages: clean pattern establishment, harness-first architecture, documentation from day one, no coordination debt, progressive complexity, and the agent-wrote-it-so-the-agent-can-read-it principle.
- Brownfield migration is possible but harder: document current state, establish constraints for new code, create safe zones, incrementally document, and progressively tighten constraints.
- The factory metaphor: invest in building the factory before producing widgets. The factory (harness) pays for itself quickly when the machines (agents) run at high throughput.
- Factories need maintenance — the harness evolves over time, driven by observed bottlenecks. Automate maintenance through garbage collection agents that run on a regular cadence.
